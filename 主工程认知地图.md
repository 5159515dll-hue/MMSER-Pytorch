# 主工程认知地图

这份文档回答一个很实际的问题：

- 对这个仓库，哪些部分你必须真正吃透，才算对工程有明确认知
- 哪些部分你只需要知道“它是干什么的”，不必一开始钻进去

本文只讨论当前主工程，也就是根目录的 `manifest + gpu_stream` 主线，不把 `legacy/baseline_v1/` 当成默认阅读对象。

## 先建立总判断

如果你现在要对这个仓库形成“明确认知”，你首先必须接受 3 个边界：

1. 当前唯一正式主线是根目录 `gpu_stream` 路线。
2. `legacy/baseline_v1/` 是历史归档，不是你理解主工程时的中心。
3. 这个仓库不是“一个训练脚本”，而是一条完整链路：
   - 数据整理
   - manifest 构建
   - 媒体准备
   - 可选 input cache
   - 训练
   - 推理
   - 多 seed 汇总
   - 论文级协议约束

如果这三个边界没有立住，后面很容易把旧路线、主线、缓存路线和论文口径混在一起。

## 你必须真正吃透的部分

下面这些模块，不是“知道名字”就够了，而是建议你读到能自己复述它们在整条链路中的作用。

### 1. 主线入口与总流程

你必须吃透：

- [train.py](/Users/dailulu/projects/MMSER-Pytorch/train.py)
- [batch_inference.py](/Users/dailulu/projects/MMSER-Pytorch/batch_inference.py)
- [gpu_stream_train.py](/Users/dailulu/projects/MMSER-Pytorch/gpu_stream_train.py)
- [gpu_stream_infer.py](/Users/dailulu/projects/MMSER-Pytorch/gpu_stream_infer.py)

你需要真正明白：

- 训练是怎么从 `train.py` 进入 `gpu_stream_train.py` 的
- 推理是怎么从 `batch_inference.py` 进入 `gpu_stream_infer.py` 的
- 训练时哪些东西会被写入 `metrics.json`、`checkpoints/best.pt`
- 推理时哪些东西会被写入 `inference_val.metrics.json`、`inference_test.metrics.json`

如果你连“训练主函数”和“推理主函数”都没吃透，就不能说你真正理解了工程。

### 2. manifest 驱动的数据定义

你必须吃透：

- [build_split_manifest.py](/Users/dailulu/projects/MMSER-Pytorch/build_split_manifest.py)
- [manifest_utils.py](/Users/dailulu/projects/MMSER-Pytorch/manifest_utils.py)
- [filter_meld_manifest.py](/Users/dailulu/projects/MMSER-Pytorch/filter_meld_manifest.py)

你需要真正明白：

- 一个 manifest item 至少包含什么字段
- `label_idx`、`label_en`、`speaker_id`、`video_path`、`audio_path`、`mn`、`masked_mn` 分别是干什么的
- `manifest_sha256` 为什么重要
- `validity.scientific_validity`、`claim_scope` 为什么会影响论文表述

如果你不理解 manifest，你后面对缓存、训练、推理和论文级可信的理解都会是散的。

### 3. 数据入口和缓存入口

你必须吃透：

- [data.py](/Users/dailulu/projects/MMSER-Pytorch/data.py)
- [prepare_dataset_media.py](/Users/dailulu/projects/MMSER-Pytorch/prepare_dataset_media.py)
- [input_cache.py](/Users/dailulu/projects/MMSER-Pytorch/input_cache.py)
- [build_mainline_input_cache.py](/Users/dailulu/projects/MMSER-Pytorch/build_mainline_input_cache.py)

你需要真正明白：

如果你不懂这一层，你就会误以为“缓存只是提速工程细节”，但在这个仓库里，缓存其实已经被纳入正式实验契约。

## 到什么程度，才算你对这个工程有明确认知

如果你已经能不看源码，直接回答下面这些问题，就可以说你对主工程有明确认知：

1. 这仓库当前正式主线是什么，旧主线是什么。
2. 一个样本从原始 MP4 / metadata 到 manifest item，会经过哪些步骤。
3. 一个 manifest item 怎么变成训练时的 batch。
4. 音频、文本、RGB、flow、prosody 各自在哪一层进入模型。
5. `best.pt` 是怎么被选出来的，为什么不能用 test 选 checkpoint。
6. `input_cache` 缓存的到底是什么，为什么不能缓存 embedding。
7. `paper_grade.eligible`、`claim_scope`、`scientific_validity` 分别控制什么。
8. 为什么正式结果不能只报单次最好 seed。
9. 为什么第四组不是旧 CPU optical flow 的复现。
10. 为什么同一组实验不能混用不同 manifest、不同 cache contract。

只要这 10 个问题你答不稳，就说明你还没有真正吃透主工程。

## 你现阶段可以先不追求的东西

下面这些不必一开始追求“全部会背”：

- 所有命令行参数的每个细节
- 所有兼容层的每一行实现
- 旧 baseline 的所有结构
- 每个 audit 脚本的全部实现
- 每个 utility 的全部边界情况

但你必须知道它们各自处在什么位置、为什么存在、什么时候需要回来读。

## 最后一句判断

对这个工程形成明确认知，不等于“你把所有文件都读完了”。

真正的标准是：

- 你知道主线是谁
- 你知道数据怎么进来
- 你知道 batch 怎么形成
- 你知道模型怎么吃多模态
- 你知道结果怎么被认定为正式论文结果
- 你知道哪些东西虽然存在，但不是主线核心

如果这六件事已经立住，你就不是“会跑命令”，而是真的对工程有了结构化认知。
